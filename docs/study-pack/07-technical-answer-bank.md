# 07 AI Agent Engineer 技术问答库

这份文档是学习用的答案库，不是项目需求文档。使用方式：

1. 先按主题学，不要按题号散着背。
2. 每一类先掌握“核心模型”，再背具体问法。
3. 回答时尽量绑定 Agent Forge 里的代码证据：`AgentLoop`、`ContextStrategy`、`ToolRegistry`、`StepController`、`SupervisorAgent`、`TraceRecorder`。
4. 不属于本项目的题，比如 GraphRAG、多模态、训练，不要硬说项目做了；要说清楚它属于延展知识。

## 系统学习方法

### 学习主线

把所有题归成 7 条主线：

1. Agent runtime：Agent 是怎么循环、怎么停、怎么恢复的。
2. Context engineering：模型每轮到底看什么，为什么看这些。
3. Tool governance：工具怎么注册、选择、校验、执行、审计。
4. Multi-agent orchestration：多个 Agent 如何分工、通信、验证。
5. Safety and control：权限、审批、沙箱、回放、风险控制。
6. Evaluation and observability：怎么证明 Agent 有用，怎么定位问题。
7. Adjacent knowledge：RAG、GraphRAG、多模态、训练、C 端产品。

### 学习渠道

不要只看问答。建议按这个顺序学：

1. 读自己的项目代码：先看 `cli.py`、`agent_loop.py`、`context_strategy.py`、`registry.py`、`control.py`。
2. 跑自己的 trace：`python run_demo.py --mode single --trace-file trace-single.json`，看每个 event。
3. 读开源项目源码：重点看 coding agent 的 context、tool、permission、diff、session。
4. 读论文只看方法框架：ReAct、ReWOO、SWE-agent、Tree of Thoughts、GraphRAG。
5. 用 badcase 学：每次 agent 失败，按“上下文错、工具错、权限错、规划错、验证错、模型错”分类。
6. 建自己的答题模板：每题都按“定义 -> 工程实现 -> 风险 -> 我的项目证据 -> 生产扩展”回答。

### 答题模板

通用结构：

```text
先给定义。
再讲工程实现。
再讲风险和取舍。
最后绑定我的项目或生产扩展。
```

示例：

> ReAct 不是让模型无限思考，而是把推理和工具执行变成受控循环。工程上每轮包括 context assembly、LLM call、tool call、permission check、tool execution、observation feedback、recovery decision。我的项目里 `AgentLoop` 实现循环，`StepController` 控制失败、重复调用和预算，`TraceRecorder` 记录每步证据。

## 1. Agent 架构与项目设计

### 1.1 介绍一个 AI Agent 项目，整体架构怎么设计？

答：我会介绍 Agent Forge。它不是简单调用 LLM，而是一个 CodingAgent runtime：CLI 接收任务，ContextStrategy 组装上下文，ModelGateway 调模型，AgentLoop 执行 ReAct 循环，ToolRegistry 管工具，Safety 层做权限和沙箱，Trace/Session 记录审计，SupervisorAgent 支持多 Agent 编排。核心是让模型负责推理，让 runtime 负责边界、状态、工具和验证。

### 1.2 为什么选择 Workflow，而不是完全自主决策的 Agent？

答：完全自主 Agent 能力强但风险高，容易越权、死循环、误操作。Workflow 可控、稳定、可审计，适合确定性强的业务链路。我的做法是混合：关键流程用 workflow/supervisor 控制，局部需要读环境、处理失败、选择工具的节点用 ReAct。

### 1.3 Workflow 里哪些节点需要 ReAct 能力？

答：需要动态观察环境的节点适合 ReAct，比如代码定位、读取文件、修复测试失败、根据 tool observation 调整计划。纯规则节点，比如权限校验、状态流转、预算判断、审批策略，不需要 ReAct，应该用确定性代码。

### 1.4 Tools、Workflow、Agent 的本质区别是什么？

答：Tool 是一个可执行动作，比如读文件、跑测试。Workflow 是固定流程，比如 plan -> code -> test -> review。Agent 是带状态的决策循环，它会根据上下文和 observation 决定下一步调用哪个工具。三者关系是：Agent 使用 Tools，Workflow 编排 Agent 和 Tools。

### 1.5 什么场景适合规则链路，什么场景适合大模型？

答：确定性、高风险、强事实的场景适合规则链路，比如权限、支付、活动资格、库存、审批。语义理解、开放式规划、代码分析、自然语言生成适合大模型。生产系统通常是规则托底，模型处理不确定性。

### 1.6 当前阻碍 Agent 大规模落地的最大挑战是什么？

答：不是单点模型能力，而是可控性和可验证性。Agent 会调用工具、跨多步执行，风险包括幻觉、越权、死循环、成本失控、状态不可恢复、评测困难。落地关键是 runtime：权限、sandbox、trace、eval、rollback、human-in-the-loop。

### 1.7 如何平衡 Agent 的可控性和能力？

答：用 bounded autonomy。模型可以在局部循环里选择工具和下一步动作，但工具集合、权限、预算、最大步数、输出声明、审计回放由 runtime 控制。这样既保留模型的推理能力，又把风险限制在工程边界内。

### 1.8 长链路 Agent 为什么能接受十几分钟耗时？

答：长链路 Agent 通常处理高价值任务，比如代码修复、数据分析、复杂工作流自动化。用户能接受较长耗时的前提是任务异步化、有进度、有中间状态、有失败恢复、有最终报告。不能让用户盯着无反馈的黑盒等十分钟。

### 1.9 不同业务场景下，延迟和效果如何取舍？

答：C 端对话要低延迟，优先小模型、缓存、流式输出和局部工具。后台任务和代码修复可以牺牲延迟换准确率。高风险业务要优先正确性和可审计，不能为了速度跳过验证。核心是按任务价值、风险和用户等待容忍度分层。

### 1.10 你最突出的两个架构设计是什么？为什么？

答：第一是上下文工程，把 repo map、文件排序、代码预览、memory summary、topic shift 做成独立策略层。第二是执行控制，把失败分类、重复调用、预算、权限和 recovery decision 放到 runtime。它们解决的是 Agent 落地最核心的两个问题：看什么和怎么安全地做。

### 1.11 你的 Agent 项目为什么做出现在这些决策？

答：目标是 production-style CodingAgent runtime core，所以没有把 TUI、IDE 插件、多模态和训练塞进同一个仓库，而是集中做 AgentLoop、ContextStrategy、ToolRegistry、HookManager、ExecutionEnvironment、TaskStateStore、StepController、Trace、Eval 和 Supervisor。这样能讲清 CodingAgent 的核心控制面，也保留产品外壳的扩展边界。

### 1.12 如何看待 Manus / OpenClaw 这类 Agent 项目的技术落地？

答：这类项目证明通用 Agent 有需求，但落地难点不在 demo，而在 runtime 工程：长期任务状态、权限、安全、工具稳定性、成本、评测、失败恢复。我的判断是生产级 Agent 会走“关键流程确定性 + 局部 Agent 智能化”的路线。

### 1.13 是否看过 Agent 开源项目源码？核心实现是什么？

答：看这类项目要抓主线：CLI/入口、上下文构造、工具协议、模型调用、权限审批、diff/patch、session、eval。比如 coding agent 的核心不是单次 prompt，而是如何把 repo context、tool call、observation、patch、test、trace 串起来。

### 1.14 Agent 的整体框架通常包含哪些模块？

答：通常包括输入理解、上下文管理、memory、planner、LLM gateway、tool registry、executor/sandbox、permission、state/session、trace/metrics、eval、human-in-the-loop。多 Agent 系统还需要 supervisor、task graph、artifact handoff。

### 1.15 如何设计一个智能导购 Agent？

答：导购 Agent 要拆成意图理解、商品召回、用户画像、业务规则、对话澄清、推荐排序、事实校验、下单/售后工具。模型负责理解和表达，商品价格、库存、活动资格必须走确定性 API 和规则校验。

### 1.16 智能导购 Agent 中，感知、规划、记忆、执行模块如何协同？

答：感知负责理解用户需求和上下文，记忆提供偏好和历史，规划决定先澄清还是召回商品，执行调用搜索、库存、优惠、下单等工具。执行结果再进入 observation，更新计划和回答。

### 1.17 开放式任务 Agent 如何权衡“探索新信息”和“利用已有知识”？

答：可以用预算和置信度控制。已有上下文足够时利用当前知识；信息不足、冲突或低置信度时探索新信息。工程上可设置检索次数、工具调用预算、停止条件和 verifier，避免无限探索。

### 1.18 Agent 如何支持长期开放式任务执行？

答：需要 session、持久状态、任务分解、进度记录、断点恢复、摘要压缩、异步执行和审计日志。每一步都要有可恢复状态，而不是依赖一次上下文窗口完成全部任务。

### 1.19 如何在不降低通用推理能力的前提下嵌入业务规则约束？

答：不要把所有规则塞进 prompt，而是把规则放到 tool、policy、validator、permission layer。模型仍然负责开放式推理，但动作是否允许、事实是否正确、结果是否合规由确定性系统判断。

### 1.20 复杂 Agent 闭环中，为什么仅靠 RAG 不能彻底解决幻觉？

答：RAG 只改善“信息来源”，不能保证模型正确使用信息，也不能保证工具调用正确、流程安全、输出经过验证。复杂 Agent 还需要 tool observation、事实校验、输出 guardrail、权限控制和 eval。

## 2. Planning、ReAct 与推理控制

### 2.1 ReAct 框架的工程实现细节是什么？

答：工程上 ReAct 是循环：组装上下文，模型决定 final 或 tool call，runtime 校验工具和权限，执行工具，得到 observation，再把 observation 放回下一轮上下文。关键是每轮都要有状态、trace、stop condition 和异常处理。

### 2.2 ReAct 消息格式如何设计？

答：一般是 system 放规则和上下文，user 放任务，assistant 返回 tool_calls，tool role 返回 observation，下一轮 assistant 基于 observation 继续决策。tool response 要带 tool_call_id，避免多个工具结果混淆。

### 2.3 为什么使用 ReAct？

答：因为很多任务不能一次生成完成，需要和环境交互。比如 coding agent 必须读文件、跑测试、根据失败信息修复。ReAct 把推理和行动交替起来，让模型用真实 observation 修正下一步。

### 2.4 还知道哪些 Agent 推理模式？

答：常见有 ReAct、Plan-and-Execute、ReWOO、Reflection、Tree of Thoughts、MCTS planning、workflow with agentic nodes。生产上最常用的是 workflow + ReAct，因为成本和可控性最好。

### 2.5 ReWOO 是什么？

答：ReWOO 是先规划一串工具调用，并用变量引用中间结果，然后再执行和汇总。它比 ReAct 少一些模型-工具往返，延迟可能更低，但对初始计划质量要求更高。

### 2.6 Plan-and-Execute 是什么？

答：先生成计划，再按计划执行，每步可以是工具或子 Agent。好处是结构清晰、可审计；缺点是环境变化时计划可能过时，所以执行中还需要观察和修正。

### 2.7 如何用 MCTS 做 Agent planning？

答：把每个 action 当作搜索树节点，用 rollout 评估不同路径的成功概率或价值，通过 UCB 平衡探索和利用。工程上要限制分支、深度和成本，并用 verifier 给节点打分，否则线上成本不可控。

### 2.8 推理-行动循环中，如何纠正逻辑坍缩或无效工具调用？

答：要识别重复工具、无效参数、连续失败和无进展状态。runtime 应该把失败分类，给 recovery hint，必要时停止或请求人工，而不是让模型无限重试。

### 2.9 Tree of Thoughts 能否用于线上系统？

答：可以，但不能无节制使用。ToT 适合高价值、强推理任务，不适合低延迟高频请求。线上要配分支限制、缓存、早停、verifier 和成本预算。

### 2.10 如何平衡 ToT 的成本和效果？

答：先用普通 ReAct/Plan-Execute，只有低置信度或高价值任务触发 ToT。限制候选分支数和深度，用 cheap model 生成候选，用 strong model 或规则 verifier 评估。

### 2.11 Agent 反思机制如何设计？

答：Reflection 不应该每轮都做，而应由失败、低置信度、验证不通过触发。反思的输出应该是结构化修正：失败原因、下一步动作、是否重试、是否请求人工。

### 2.12 如何避免 Reflection 过度纠正或陷入循环？

答：设置最大反思次数、重复动作检测、失败预算和验证门槛。Reflection 只能改变计划或参数，不能绕过权限和安全策略。

### 2.13 如何设计有节制、基于置信度阈值的反思触发条件？

答：触发条件可以包括工具失败、测试失败、模型自评低置信度、事实冲突、用户纠正、输出 guardrail 失败。阈值要结合任务风险，高风险场景宁愿多验证，低风险场景减少反思。

### 2.14 Agent 出现循环调用或思维死循环时怎么办？

答：检测相同 tool+args 重复、连续失败次数、无新增 observation、超过步数或时间预算。触发后停止、总结当前状态、给出失败原因，必要时转人工。

### 2.15 Token 过长导致 attention 稀释，为什么会降低 Agent 指令遵循能力？

答：上下文过长时，关键指令和当前任务在 attention 中被大量无关内容稀释，模型更容易忽略约束、引用旧信息或丢失目标。因此要做检索、摘要、截断和 attention sink。

### 2.16 标准 Attention 在多轮 Agent 对话中主要带来哪些工程问题？

答：问题包括上下文窗口有限、旧信息污染新任务、重要约束被稀释、工具结果太长、摘要丢细节。工程上需要 memory 分层、topic shift 判断、检索和压缩策略。

### 2.17 Attention Sink 是什么？

答：Attention Sink 是在长上下文里稳定保留的一组关键锚点，帮助模型维持任务目标和行为约束。我的项目里把最新任务、先观察再编辑、observation 回写、不得虚假声明测试通过作为稳定锚点。

### 2.18 在上下文不足时，如何在不丢失 Attention Sink 的前提下保持生成连贯性？

答：优先保留 system policy、当前任务、attention sink、最近 observation、关键文件摘要。压缩旧 memory、截断大文件、只保留 top relevant context，必要时请求用户补充。

## 3. Tool Calling、协议与工具治理

### 3.1 tool_response 应该用什么角色返回？为什么？

答：应该用 `tool` role，并带上对应的 `tool_call_id`。这样模型和 runtime 都能知道这个 observation 对应哪个工具调用，避免多个并发工具结果混淆。

### 3.2 tool call 和 observation 如何进入下一轮推理？

答：assistant 先产生 tool_calls，runtime 执行后把结果作为 tool message 加入 messages。下一轮 LLM 看到之前的 tool call 和 observation，再决定继续调用工具还是 final。

### 3.3 工具库有上百个工具时，如何让模型快速选对工具？

答：不能把所有工具都塞给模型。应该做工具路由：按任务意图、领域、权限、历史成功率召回候选工具，只把少量相关 schema 给模型，再用参数校验和 trace 反馈纠错。

### 3.4 tool schema 如何设计才能减少误调用？

答：schema 要短、明确、参数类型清楚、必填字段明确、枚举值受限、描述包含使用边界和反例。复杂参数要拆小，不要让模型自由拼大 JSON。

### 3.5 候选工具超过 100 个时，如何设计工具路由策略？

答：先做 coarse routing 到领域，再用关键词/embedding/规则召回 top tools，最后按权限和上下文过滤。高频工具可以缓存，危险工具默认不暴露。

### 3.6 如何解决工具检索过程中的召回偏差？

答：需要混合召回、失败反馈、工具别名、usage statistics 和人工标注 badcase。工具描述要标准化，否则 embedding 检索会偏向描述相似但能力不匹配的工具。

### 3.7 多工具高频调用任务中，如何压低端到端延迟？

答：减少模型往返、并行无依赖工具、缓存工具结果、用小模型做路由、大模型只做关键决策。工具依赖图明确后，可把独立节点并发执行。

### 3.8 多个外部工具存在依赖时，如何设计工具调用依赖图？

答：把工具调用建成 DAG，节点是工具任务，边是输入依赖。每个节点记录状态、输入、输出、重试策略、超时和幂等键。失败时只重试受影响子图。

### 3.9 多工具调用如何并行执行以最小化响应延迟？

答：识别无共享写状态、无数据依赖的工具并发执行。有共享资源或写冲突的工具必须串行或隔离执行。并发结果要合并到统一 observation。

### 3.10 MCP、A2A、Skills、Function Call 分别是什么？

答：Function Call 是模型调用函数的协议。MCP 是外部工具/资源接入协议。Skills 是高层能力包，通常包含说明、示例和工具组合。A2A 是 Agent 之间通信和协作的协议思路。

### 3.11 MCP、A2A、Skills、Function Call 有什么区别？

答：Function Call 关注模型到工具；MCP 关注工具生态标准化接入；Skills 关注能力封装和复用；A2A 关注 Agent 间任务协作。层级不同，不是互斥关系。

### 3.12 MCP 和传统 Agent Skills 有什么区别？

答：MCP 更像工具/资源连接协议，解决外部系统如何暴露能力。Skills 更像能力包或操作手册，解决模型如何使用一组能力完成任务。MCP 偏接口，Skills 偏行为组织。

### 3.13 多智能体环境中，如何动态发现并注册跨协议工具？

答：需要 tool discovery registry，把 MCP、HTTP API、本地函数等适配成统一 ToolSpec。注册时要做权限、schema、租户、版本、健康检查和审计。

### 3.14 A2A 为什么不用现成协议，而要自建一套？

答：因为 Agent 间通信不只是 RPC，还需要表达任务意图、能力声明、状态、长期上下文、handoff、拒绝、回退和审计。传统协议能承载传输，但不一定有 Agent 语义。

### 3.15 Skills 和 Tools 的区别是什么？

答：Tool 是具体动作，比如 read_file。Skill 是完成某类任务的能力组织，可能包含 prompt、步骤、工具集合、示例和约束。Tool 是原子动作，Skill 是组合能力。

## 4. Multi-Agent 与编排

### 4.1 Multi-Agent 系统如何设计三层架构？

答：可以分为 orchestration 层、agent worker 层和 tool/runtime infra 层。orchestration 负责任务图、状态、路由、权限；worker 负责角色任务；infra 提供工具、memory、trace、sandbox。

### 4.2 Agent 之间如何通信？

答：不要只靠自然语言聊天。生产系统应该用结构化 handoff、artifact、shared state 和 trace。自然语言可以作为摘要，但决策应依赖机器可读状态。

### 4.3 Agent 之间怎么编排？

答：常见方式有 supervisor、DAG workflow、blackboard/shared-state、去中心化协商。生产优先 supervisor 或 DAG，因为可控、可审计、容易设置停止条件。

### 4.4 Agent 框架选型怎么考虑？

答：看状态管理、工具协议、可观测性、human-in-loop、并发/任务图、部署复杂度、生态集成。项目可以自研 runtime core，证明你理解核心边界；生产也可以按团队生态选 LangGraph、CrewAI 或 OpenAI Agents。

### 4.5 shared-state、agent team、主子结构有什么区别？

答：shared-state 是多个 Agent 读写同一个状态空间；agent team 强调角色协作；主子结构由 supervisor 分配任务和验证结果。主子结构最可控，去中心化最灵活但风险最高。

### 4.6 如何防止 Multi-Agent 互相 A2A 停不下来？

答：设置 supervisor、最大轮数、任务图终止条件、消息预算、重复检测和人工升级。Agent 不能随意互相拉起新任务，必须经过调度策略。

### 4.7 主子模式下，子 Agent 产生幻觉怎么办？

答：子 Agent 输出不能直接信任，要用 tool observation、测试、事实 API、reviewer 或 verifier 校验。Supervisor 只接受带证据的 artifact。

### 4.8 子 Agent 的输出是否应该直接给用户？

答：通常不应该。子 Agent 输出是中间结果，需要 supervisor 汇总、去重、验证和风险过滤。直接暴露会导致口径不一致和幻觉外泄。

### 4.9 Supervisor 如何验证子 Agent 输出？

答：看结构化 artifact、工具执行结果、测试结果、diff、trace 和风险字段。必要时调用独立 verifier 或让 reviewer agent 复核。

### 4.10 Agent 之间如何传递上下文和状态？

答：通过 handoff payload、TaskArtifact、shared state、session store。传递要有边界，不能把所有上下文原样塞给下一个 Agent。

### 4.11 多 Agent 协作中，不同 Agent 的记忆如何隔离与共享？

答：每个 Agent 保留局部 memory，Supervisor 维护共享任务状态。共享只传递必要 artifact，避免一个 Agent 的无关上下文污染另一个 Agent。

### 4.12 如何避免不同工具或 Agent 之间的上下文污染？

答：用角色工具 allowlist、上下文筛选、artifact contract、topic boundary 和 trace。不同角色只看完成任务需要的最小上下文。

### 4.13 去中心化 Multi-Agent 中，如何设计任务分配协议？

答：需要能力声明、任务竞标、负载状态、优先级、冲突解决、超时和回收机制。每个 Agent 接任务前要声明能做什么、成本多少、预计多久。

### 4.14 Agent 如何根据自身能力和当前负载自主协商并接管子任务？

答：可以用 contract-net 类协议：发布任务，Agent 根据能力和负载报价，调度器选择，执行后提交 artifact。必须有超时、失败转移和审计。

## 5. Agent 执行控制与异常处理

### 5.1 Agent 出现死循环怎么办？

答：检测重复 tool+args、连续失败、无新增信息、超过最大步数或时间预算。触发后停止并输出当前状态，不让模型继续自我消耗。

### 5.2 Agent 异常处理机制如何设计？

答：先把异常分类：模型错误、工具错误、权限错误、参数错误、业务错误、超时、预算错误。每类定义 retryable、恢复策略、是否升级人工。

### 5.3 Agent 工具调用失败如何处理？

答：工具失败要变成 observation，进入下一轮推理。runtime 根据失败类型决定重试、修正参数、换工具、停止或请求人工。

### 5.4 支付接口超时这类工具失败如何处理？

答：支付是非幂等高风险操作，不能盲目重试。要用幂等键、订单状态查询、超时补偿、事务日志和人工介入策略。先查状态再决定是否重试。

### 5.5 如何区分可重试失败和不可重试失败？

答：临时网络、超时、参数可修复、patch anchor mismatch 通常可重试。权限拒绝、越权、重复非幂等动作、用户拒绝、业务规则不允许通常不可重试。

### 5.6 Agent 如何做超时控制？

答：分三层：单工具 timeout、单模型请求 timeout、整个任务 timeout。超时后记录状态，必要时保存 session，允许恢复或人工接管。

### 5.7 Agent 如何做最大步数控制？

答：每轮 ReAct 算一步，超过 max_steps 停止并总结已完成/未完成内容。不同任务和角色可以有不同步数预算。

### 5.8 Agent 如何做成本预算控制？

答：记录 prompt/completion token、模型单价、工具成本和总调用次数。超过预算前降级模型、减少检索、停止任务或请求用户确认。

### 5.9 Agent 决策错误导致数据误删，系统如何防范？

答：危险操作默认 deny 或 ask，写操作先 dry-run/diff，关键数据做权限隔离和备份。执行后要有审计和 rollback。

### 5.10 高风险操作是否需要二次确认？

答：需要。涉及写文件、删除、支付、发消息、外部提交、权限变更等操作都应该 human-in-the-loop 或至少二次确认。

### 5.11 Agent 如何设计权限、审计、回放机制？

答：权限由 policy 决定，工具执行写 trace，session 保存输入、动作、observation、diff、metrics。回放时根据 trace 复现决策链，rollback 根据快照恢复。

### 5.12 Agent 如何避免越权调用工具？

答：给不同 Agent 角色配置工具 allowlist，工具内部再做 sandbox 和权限检查。模型看不到或调用不了不属于它的工具。

### 5.13 Agent 连续执行网页操作或 API 调用时，中间失败如何回滚和重试？

答：要把流程拆成可恢复状态机，每步记录外部状态和幂等键。可回滚操作做补偿，不可回滚操作做状态查询和人工处理。

### 5.14 如何避免重复执行非幂等操作？

答：使用 idempotency key、操作去重表、状态查询、重复动作检测和审批。非幂等操作不能只靠 LLM 判断是否重复。

### 5.15 任务执行超过单次 Token 限制时，如何支持断点续写或继续生成？

答：保存 session、任务状态、摘要、已完成 artifact、下一步计划。恢复时加载 summary 和必要证据，而不是完整历史。

### 5.16 Agent 执行动作前，如何做输入输出一致性的自我验证？

答：动作前校验 schema、权限、路径、业务规则；动作后校验 observation、测试、状态变化和输出声明。高风险动作还要 dry-run。

### 5.17 如何设计 Self-Verification 模块减少工具参数错误和推理链断裂？

答：Self-Verification 应检查工具名、必填参数、参数类型、上下文依据、风险等级和预期 observation。失败时返回结构化错误，让 Agent 修正而不是直接执行。

## 6. Memory、Context 与 Session 管理

### 6.1 Agent 的记忆机制怎么设计？

答：分为短期 memory、summary memory、长期 memory 和 session state。短期存最近 observation，summary 存压缩历史，长期存稳定事实或用户偏好，session 存任务执行状态。

### 6.2 短期记忆和长期记忆分别如何实现？

答：短期记忆通常存在当前上下文或内存队列，随任务结束释放。长期记忆存在数据库/向量库/画像系统，按需要检索加载，必须有更新和冲突治理。

### 6.3 memory 为什么要分短期、长期、summary？

答：因为信息生命周期不同。短期信息细节多但很快过期，长期信息稳定但要谨慎写入，summary 用来压缩长历史，避免上下文爆炸。

### 6.4 长期 memory 存什么？

答：存稳定、可复用、经过验证的信息，比如用户偏好、项目约定、业务规则、历史决策。临时推测、未验证事实和一次性工具结果不应该写长期 memory。

### 6.5 长期 memory 什么时候加载？

答：当当前任务与历史主题相关、用户有指代、需要偏好/规则/项目约定时加载。topic shift 或低相关时不要加载，避免污染上下文。

### 6.6 memory 是模型自己选，还是规则触发？

答：生产上应该规则和模型结合。候选 memory 可以由检索/规则召回，模型可参与判断相关性，但最终注入要受 token budget、隐私和安全策略控制。

### 6.7 上下文超出限制时如何处理？

答：保留系统规则、当前任务、关键 observation、相关证据。压缩旧历史，截断大文档，检索 topK，必要时请求用户澄清或分阶段执行。

### 6.8 滑动窗口和动态摘要的区别是什么？

答：滑动窗口保留最近 N 条，简单但会丢早期关键事实。动态摘要把旧历史压缩成结构化信息，能保留长期目标，但可能丢细节，所以最好保留证据引用。

### 6.9 长上下文对话中，如何让 Agent 不忘记关键信息？

答：使用 attention sink、任务状态摘要、关键事实表、retrieval memory 和显式计划。不要依赖模型自己在长上下文里记住所有内容。

### 6.10 除了向量检索，还有什么记忆方法？

答：关键词/BM25、结构化数据库、知识图谱、规则表、用户画像、session state、摘要树、事件日志。不同 memory 类型适合不同查询。

### 6.11 多轮对话里，指代不清怎么处理？

答：先判断是否能从当前 session 安全消解，不能就反问。高风险动作不能靠猜测，必须确认对象、时间、范围和操作。

### 6.12 同一 session 中用户频繁切换话题，如何处理上下文？

答：做 topic segmentation。相关话题局部继承，不相关话题新建上下文或降低历史权重。旧任务未完成时要暂停或确认是否切换。

### 6.13 上一秒问游戏，下一秒问经济，如何判断是否继承上下文？

答：看实体、意图、领域和用户指代。如果没有明确关联，不继承游戏上下文，只保留通用系统规则。

### 6.14 用户先问原神，再问崩铁，如何处理上下文？

答：同属游戏领域但实体不同，可以继承用户表达偏好和对比意图，但不能继承具体事实。最好识别为 related topic，而不是 same topic。

### 6.15 用户说“按老样子帮我订一下”，Agent 如何处理？

答：需要读取长期偏好和历史订单，但下单前必须确认关键参数，比如时间、地点、商品、价格。不能直接执行高风险交易。

### 6.16 Agent 知识闭环中，哪些信息进入向量库，哪些进入上下文窗口，哪些转为模型权重记忆？

答：向量库存可检索外部知识和历史文档；上下文窗口放当前任务必需证据；模型权重只适合通过训练学习通用模式，不适合频繁变化或个人隐私信息。

### 6.17 对话轮数很多且上下文窗口不足时，如何缓解信息遗忘？

答：分层 memory、动态摘要、关键事实表、检索式加载、任务状态机。每轮只加载当前决策需要的信息。

### 6.18 摘要总结容易丢失关键细节，长文本 Agent 如何处理？

答：摘要要结构化并保留证据指针，比如原文位置、文件路径、时间戳。关键细节不要只存在摘要里，需要可回查原始证据。

### 6.19 短期记忆和长期记忆并存时，上下文超限如何选择保留、压缩或丢弃？

答：优先保留当前任务、约束、最新失败、强相关长期事实。低相关旧对话压缩或丢弃，冲突信息保留来源和置信度。

### 6.20 请给出至少两种记忆管理策略。

答：第一，滑动窗口 + 动态摘要。第二，长期 memory 检索 + 当前任务状态表。还可以用实体记忆、事件日志和知识图谱。

### 6.21 从非结构化文本更新长期记忆时，如何避免虚假或矛盾信息写入？

答：先抽取候选事实，再做置信度评估、来源记录、冲突检测和用户确认。未验证事实只进临时 memory，不进长期 memory。

### 6.22 如何设计信息置信度评估与冲突解决流程？

答：记录来源、时间、证据、抽取模型置信度和用户确认状态。冲突时按权威来源、新鲜度、用户确认优先级解决，不能静默覆盖。

### 6.23 多轮对话任务中，用户中途改需求，如何做意图回溯和计划修正？

答：识别新需求影响哪些已计划/已执行步骤，更新任务状态，取消过时子任务，必要时回滚或重新确认。不能继续执行旧计划。

### 6.24 如何避免执行过时或无用子任务？

答：每个子任务执行前检查当前目标版本、依赖状态和用户最新指令。任务图节点要能 skipped/cancelled，而不是只会 pending/running/done。

## 7. RAG、GraphRAG 与知识检索

### 7.1 RAG 从文档入库到线上检索，完整链路怎么做？

答：链路包括文档采集、解析、清洗、切分、embedding、索引、metadata、召回、rerank、上下文拼装、生成、引用和后校验。线上还要做版本、权限、更新、监控和 badcase 回流。

### 7.2 如何做 RAG？

答：先把知识源处理成可检索 chunk，建立关键词/向量索引；请求进来后做 query rewrite、召回、rerank、压缩上下文，再让模型基于证据回答。最后做引用和事实校验。

### 7.3 为什么要混合检索？

答：关键词检索适合精确词、编号、专有名词；向量检索适合语义相似和表达变化。混合检索能提高召回稳定性，减少单一路径偏差。

### 7.4 向量检索和关键词检索分别解决什么问题？

答：向量检索解决语义匹配，关键词检索解决精确匹配。比如代码函数名、订单号、政策条款编号更适合关键词；用户自然语言描述更适合向量。

### 7.5 BM25 和向量检索如何融合？

答：常见做法是分别召回 topK，然后用加权分数、RRF 或 reranker 合并。线上一般先扩大召回，再用 cross-encoder/LLM rerank 精排。

### 7.6 BM 算法是什么？

答：通常指 BM25，是基于词频、逆文档频率和文档长度归一化的文本相关性算法。它不理解语义，但对关键词匹配稳定、可解释、速度快。

### 7.7 RAG 为什么要引入父子索引？

答：子 chunk 适合精确召回，父文档适合提供完整上下文。先用小块找相关位置，再把父段落或父文档片段放进上下文，可以兼顾召回精度和语义完整性。

### 7.8 父子索引分别解决什么问题？

答：子索引解决定位精确，父索引解决上下文完整。只用子块容易丢上下文，只用父块又容易召回粗糙、浪费 token。

### 7.9 TopK 怎么定？

答：TopK 取决于问题复杂度、chunk 粒度、rerank 能力和上下文预算。一般先召回较大 K，再 rerank 压到能放进上下文的小 K。

### 7.10 召回越多一定越好吗？

答：不是。召回太少会漏信息，太多会引入噪声和 attention 稀释。RAG 的关键是高质量证据，而不是把所有内容塞给模型。

### 7.11 chunk 切分为什么不能只用固定长度？

答：固定长度会切断语义结构，比如标题、表格、代码函数、政策条款。更好的方式是按结构切分，再用长度约束兜底。

### 7.12 长文档如何处理？

答：先结构化解析目录、标题、段落、表格，再分层索引。查询时先定位章节，再取局部上下文，必要时做 map-reduce summary。

### 7.13 长文档分块时，chunk size 如何确定？

答：看文档结构、任务类型和模型窗口。问答型 chunk 可以小一些，综合总结要大一些。要用召回率、答案正确率和上下文噪声评估。

### 7.14 如何衡量分块效果好坏？

答：看 retrieval recall、MRR、nDCG、answer accuracy、引用命中率和人工 badcase。好 chunk 应该语义完整、长度适中、metadata 清楚。

### 7.15 分块后如何用聚类、Gini 系数或熵评估纯度？

答：可以看 chunk 内 embedding 分布是否集中、主题标签是否单一、实体/关键词分布是否过散。Gini 或熵能反映一个 chunk 是否混入太多主题。

### 7.16 复杂 PDF 解析最难的点是什么？

答：难点是版面结构、表格、跨页、脚注、图片文字、阅读顺序和公式。解析错了，后续 embedding 和检索都会错。

### 7.17 metadata 除了存字段，还能如何参与召回？

答：metadata 可以做过滤、加权、排序、权限控制、版本控制和 freshness 控制。例如部门、时间、文档类型、章节、实体都能影响召回。

### 7.18 文档版本机制怎么设计？

答：每个文档有 doc_id、version、effective_time、source 和 hash。检索时默认取最新有效版本，历史回答要能追溯当时版本。

### 7.19 文档有效期怎么设计？

答：给文档或 chunk 设置生效时间、失效时间和更新频率。过期内容不能直接进入回答，除非用户明确问历史信息。

### 7.20 文档更新机制怎么设计？

答：支持增量更新、hash diff、删除同步、索引重建和回滚。更新后要跑抽样评测，避免新文档污染检索质量。

### 7.21 离线文档和实时接口分别适合什么场景？

答：稳定知识适合离线文档，比如政策、手册。强实时事实适合 API，比如库存、价格、订单状态。两者冲突时通常实时接口优先。

### 7.22 生成后校验层解决什么问题？

答：解决模型错误使用证据、编造、引用不一致的问题。校验层检查答案是否被证据支持、事实是否来自权威源、是否违反规则。

### 7.23 Elasticsearch 在 RAG 系统中的作用是什么？

答：ES 适合关键词检索、过滤、聚合、高亮和权限过滤。它通常和向量库或向量插件一起做 hybrid retrieval。

### 7.24 如何优化 Elasticsearch 检索性能？

答：优化 mapping、分片、倒排字段、过滤条件、缓存、分页方式和查询 DSL。高频查询要缓存，复杂聚合要控制范围。

### 7.25 为什么采用 GraphRAG？

答：当问题需要多跳关系、实体关联、路径推理时，普通向量检索不够。GraphRAG 用实体和关系组织知识，更适合复杂关联查询。

### 7.26 GraphRAG 在复杂关联查询中的优势是什么？

答：它能沿关系边查找多跳路径，解释实体之间为什么相关。比如人、公司、职位、项目、交易关系，图结构比纯文本相似度更直接。

### 7.27 图谱的好处是什么？

答：图谱能表达实体、关系、路径和约束，适合关联推理、推荐、风控、搜索增强。它可解释性强，但构建和维护成本高。

### 7.28 知识图谱如何构建？

答：流程是定义 schema、本体和关系类型；做 NER、实体链接、关系抽取；清洗去重；入图库；做质量评估和增量更新。

### 7.29 什么时候选择向量数据库，什么时候选择关系型数据库？

答：语义相似检索选向量数据库；强结构化事务、精确查询、约束一致性选关系型数据库。很多系统两者配合使用。

### 7.30 向量数据库和关系型数据库如何配合？

答：向量库召回候选，关系库存结构化事实、权限、状态和业务字段。召回后用关系库过滤和补全事实。

### 7.31 图数据库中的 NER 识别怎么做？

答：NER 先从文本中识别人名、组织、地点、产品等实体，再做实体链接映射到图谱节点。可以用规则、模型或 LLM 辅助抽取。

### 7.32 如何衡量 NER 效果好坏？

答：常用 precision、recall、F1，另外看实体链接准确率、歧义消解准确率和业务关键实体召回率。

### 7.33 基于实体关联构图时，是否考虑跳转构图？

答：要考虑。只连直接关系可能漏掉隐含关联，多跳构图能增强发现能力，但也会带来噪声和路径爆炸，需要限制跳数和关系类型。

### 7.34 LinkedIn Search 这类图谱检索为什么适合图结构？

答：因为人、公司、职位、学校、技能、连接关系天然是图。搜索不仅看文本匹配，还看关系距离、共同连接、路径和权重。

## 8. Agent Eval 与迭代

### 8.1 Agent 评测体系怎么设计？

答：要分层评测：单工具、单节点、完整任务、安全、成本、延迟、用户体验。既要自动化 case，也要 badcase 回流和人工评审。

### 8.2 如何量化评估一个上线 Agent 的好坏？

答：看任务完成率、工具成功率、平均成本、延迟、用户满意度、安全违规率、人工接管率、重试率和长期留存。

### 8.3 除了准确率，还有哪些评估指标？

答：包括 task success、tool precision、latency、cost、step count、rollback rate、hallucination rate、policy violation、explainability、user correction rate。

### 8.4 用户行为能否作为效果评估信号？

答：可以，但不能单独使用。点赞、停留、复用、撤销、投诉、人工接管都能作为弱信号，需要结合任务日志和人工标注。

### 8.5 badcase 怎么回流到评测集？

答：线上 badcase 先脱敏，标注失败类型和期望行为，再加入回归集。每次改 prompt、tool、模型或策略都跑回归。

### 8.6 badcase 如何用于后续迭代？

答：先分类根因：上下文、工具、权限、模型、业务规则、评测缺失。然后对应修 retrieval、schema、policy、prompt、model 或 workflow。

### 8.7 如何评估工具调用成功率？

答：统计 tool call 总数、成功数、失败数、失败类型、重试后成功率。还要看工具是否“选对”，不是只看接口返回 200。

### 8.8 如何评估任务完成率？

答：定义任务级 success criteria，比如测试通过、用户确认、状态更新成功。不能只看模型 final answer，要看外部验证。

### 8.9 如何评估 Agent 的成本和延迟？

答：记录模型 token、模型调用次数、工具调用耗时、总任务耗时、并发等待时间。按任务类型看 P50/P95/P99。

### 8.10 如何评估 Agent 的安全违规率？

答：统计越权调用、敏感数据访问、高风险动作未审批、危险命令、虚假声明、用户投诉。安全指标要单独设阈值，不能被平均效果掩盖。

### 8.11 如何评估开放式 Agent 的性能？

答：用多维 rubric：任务完成度、过程合理性、证据充分性、安全、成本、用户满意度。可以用人工 + LLM judge，但 judge 也要抽检。

### 8.12 传统准确率和召回率为什么不够？

答：Agent 是多步执行系统，可能答案对但过程危险，也可能部分完成。需要评估过程、工具、成本、状态和安全，不只是最终文本。

### 8.13 如何设计多维度评估指标，包括任务完成度、资源效率、鲁棒性和可解释性？

答：任务完成度看 success rate，资源效率看 token/latency/tool count，鲁棒性看失败恢复和 badcase，通过可解释性看 trace 完整度和证据引用。

### 8.14 如何衡量 Agent 的 Planning 能力和 Hallucination Rate？

答：Planning 看计划可执行性、依赖正确性、步骤冗余和调整能力。Hallucination 看无证据声明、错误事实、虚假工具结果和未验证成功声明。

### 8.15 请列举具体量化评估指标或自动化评估框架。

答：指标包括 task_success、test_pass、tool_success_rate、avg_steps、avg_cost、P95 latency、safety_violation_rate。框架可以参考 SWE-bench、AgentBench、OpenAI Evals、内部 case runner。

## 9. Agent 安全、可信与可控性

### 9.1 Agent 如何做权限控制？

答：权限控制应在 runtime，不在 prompt。按角色、工具、资源、动作类型做 allow/ask/deny，工具内部再做 sandbox 和业务权限校验。

### 9.2 Agent 如何做工具调用审批？

答：高风险工具调用前生成审批事件，展示动作、参数、影响范围和原因。用户/系统审批后才执行，审批结果进入 trace。

### 9.3 Agent 如何做高风险操作拦截？

答：通过 command allowlist、路径 sandbox、敏感文件检测、业务规则、权限策略拦截。危险操作默认 deny，不让模型说服系统执行。

### 9.4 Agent 如何做 human-in-the-loop？

答：在不确定、高风险、权限不足、事实冲突时请求人工。人工输入要成为 observation，进入后续决策和 trace。

### 9.5 Agent 如何做 dry-run？

答：先生成计划、diff 或 API 请求预览，不实际执行。用户确认后再 apply。写文件、发消息、支付、删除都适合 dry-run。

### 9.6 Agent 如何做审计日志？

答：记录 task、context、model、tool call、arguments、observation、permission、approval、final answer、metrics。日志要能追溯谁在什么时候做了什么。

### 9.7 Agent 如何做执行回放？

答：用 trace 重建每一步输入、模型输出、工具结果和状态变化。回放不一定重新执行外部副作用，而是复盘决策链。

### 9.8 Agent 如何避免越权访问数据？

答：路径 sandbox、租户隔离、工具 allowlist、权限 token、数据脱敏、审计。模型不能直接拿底层凭证访问所有数据。

### 9.9 Agent 如何降低幻觉导致的业务风险？

答：强事实走 API，关键输出要引用证据，高风险决策要 verifier 或人工审批。模型负责语言和推理，不负责最终事实裁决。

### 9.10 如何设计关键节点确定性、局部节点智能化的 Agent 架构？

答：把流程、权限、状态、审批、结算做成确定性节点；把意图理解、信息检索、方案生成放到智能节点。这样既可控又有弹性。

### 9.11 强事实性场景中，如何治理模型幻觉？

答：事实来源必须是数据库/API/检索证据，模型输出必须可追溯。没有证据时反问或拒答，不能自由编造。

### 9.12 电商场景中，模型幻觉导致活动资格误判如何防范？

答：活动资格由规则引擎或权益系统判断，模型只解释结果。下单前再次校验价格、库存、资格和用户确认。

### 9.13 如何设计主动澄清决策逻辑？

答：当缺少必要参数、候选过多、风险高、上下文冲突或置信度低时主动澄清。澄清问题要少而关键，不要把所有字段一次性抛给用户。

### 9.14 什么情况下 Agent 应该反问用户？

答：目标不明确、对象不明确、权限不足、高风险动作、多个解释都合理、无法验证事实时应该反问。

### 9.15 什么情况下 Agent 可以结合历史画像强行推断？

答：低风险、可撤销、历史偏好稳定、用户表达明确省略时可以推断。高风险和交易类动作不能强行推断，必须确认。

### 9.16 面对高度模糊的电商或导购需求，Agent 如何精准理解用户意图？

答：先识别场景、预算、品类、约束、使用目的，再通过澄清和推荐引导缩小范围。结合用户画像但要避免过度假设。

### 9.17 Agent 如何遵循企业安全策略、法律法规等动态规则？

答：把规则做成独立 policy/rule service，运行时查询或热更新。模型只看到必要摘要，最终执行由 policy 决定。

### 9.18 规则约束层如何动态更新？

答：规则版本化、灰度发布、回滚、测试集验证。更新后 trace 要记录使用的规则版本。

### 9.19 可动态更新规则如何注入 Agent？

答：可以注入到 system prompt、tool schema、policy engine 或 validator。高风险规则应放 runtime policy，不应只靠 prompt。

### 9.20 如何比较不同规则注入方式？

答：prompt 注入灵活但不可靠；tool schema 能减少误调用；policy engine 最可控；validator 能兜底输出。生产上通常组合使用。

## 10. C 端对话 Agent 产品工程

### 10.1 如果负责豆包，如何保证 session 内上下文连贯？

答：做 session state、topic segmentation、memory summary、用户画像和当前任务状态。相关上下文继承，不相关话题隔离。

### 10.2 用户同一 session 中话题跳跃，如何判断是否继承历史？

答：看领域、实体、意图、指代和时间连续性。无关联时不继承，只保留通用偏好和系统规则。

### 10.3 用户上一秒问游戏，下一秒问经济，消息如何处理？

答：判断为 topic shift，经济问题不继承游戏内容。除非用户明确做类比，否则旧上下文降权或丢弃。

### 10.4 用户问原神后又问崩铁，如何处理上下文？

答：同领域但不同实体，可以继承“游戏讨论”语境和用户偏好，不继承具体角色/机制事实。必要时澄清是对比还是新问题。

### 10.5 图片生成和视频生成耗时较长，如何减少用户等待？

答：异步任务、进度展示、预估时间、先返回草稿/低清预览、完成通知。失败时给重试和原因，不让用户空等。

### 10.6 长耗时任务如何设计异步处理？

答：请求创建 job，返回 job_id，后台队列执行，前端轮询或推送进度。任务状态包括 pending/running/succeeded/failed/cancelled。

### 10.7 长耗时任务如何展示进度？

答：展示阶段进度，而不是假百分比。比如“解析输入、生成草图、渲染、审核、上传”。每阶段有可解释状态。

### 10.8 长耗时任务失败后如何处理？

答：保留失败状态、错误原因、中间产物和可重试入口。可恢复失败自动重试，不可恢复失败给用户明确下一步。

### 10.9 C 端 Agent 如何平衡体验、成本和效果？

答：按任务分层。普通聊天用快模型，高价值任务用强模型；能缓存就缓存，能异步就异步；高风险任务宁可慢也要准。

### 10.10 意图识别怎么做？

答：可以用规则、分类模型、LLM routing、embedding 检索或混合方案。输出应包括 intent、slots、confidence 和是否需要澄清。

### 10.11 为什么选择当前模型？

答：看效果、成本、延迟、上下文长度、工具调用能力、稳定性和合规。生产通常是多模型路由，而不是一个模型打所有场景。

### 10.12 模型如何优化？

答：先用 prompt/tool/context 优化，仍不够再做 SFT、偏好优化、蒸馏或 RL。优化要有 eval，不然容易只优化个别样例。

## 11. Agent Infra / 工程落地

### 11.1 Agent 系统中如何设计异步任务？

答：任务进入队列，worker 执行，状态持久化，前端通过 job_id 查询进度。支持取消、重试、超时和失败恢复。

### 11.2 Agent 系统中如何设计状态管理？

答：状态要包括 task、plan、messages、tool results、artifacts、budget、status、error。状态必须可持久化和可恢复。

### 11.3 Agent 系统中如何设计任务队列？

答：按优先级、租户、任务类型和资源隔离建队列。队列要支持重试、死信、超时、幂等和限流。

### 11.4 Agent 系统中如何设计工具网关？

答：工具网关统一注册、schema、鉴权、限流、超时、重试、审计和结果标准化。Agent 不直接调用外部系统。

### 11.5 Agent 系统中如何设计沙箱环境？

答：文件/命令/网络/权限隔离。轻量项目可用 workspace sandbox，生产 coding agent 通常需要容器、seccomp、网络隔离和资源限制。

### 11.6 Agent 系统中如何设计 tracing？

答：每步记录 agent、step、event_type、input summary、tool call、observation、latency、error、model usage。trace 要能关联 session 和用户请求。

### 11.7 Agent 系统中如何记录 tool call log？

答：记录工具名、参数、调用时间、耗时、返回状态、错误、权限结果和 trace id。敏感参数要脱敏。

### 11.8 Agent 系统中如何记录 handoff log？

答：记录 from_agent、to_agent、原因、payload、artifact、状态和时间。handoff log 用来定位 multi-agent 责任边界。

### 11.9 Agent 系统中如何做运行时观测？

答：看任务成功率、失败类型、工具耗时、模型耗时、token 成本、队列延迟、重试、人工接管和安全拦截。

### 11.10 Agent 系统中如何做线上故障定位？

答：从 trace 看最近失败节点，再看 tool log、model response、context、policy、external API。按根因分类：上下文、模型、工具、权限、业务规则、基础设施。

### 11.11 包含 3 个以上工具调用且高频请求的任务，如何降低系统端到端延迟？

答：减少模型轮次，并行独立工具，缓存工具结果，小模型路由，大模型只处理关键决策。把串行 ReAct 改为计划后批量执行能显著降延迟。

### 11.12 开放式 Agent 执行任务时，如何动态调节新信息探索和已有知识利用比例？

答：根据置信度、失败次数、信息新鲜度和预算调整。高置信度利用已有知识，低置信度或冲突时探索新信息。

### 11.13 Agent 多轮执行任务如何保存可恢复状态？

答：每轮保存 messages、summary、工具结果、artifact、当前计划、预算和状态。恢复时加载摘要和必要证据继续执行。

### 11.14 Agent 执行过程如何支持断点、恢复和重放？

答：断点靠持久 session，恢复靠 summary/artifact，重放靠 trace。外部副作用不能随意重放，要区分 replay for audit 和 re-execute。

## 12. 多模态 Agent

### 12.1 多模态大模型的具体结构是什么？

答：常见结构是视觉编码器把图像/视频转成 embedding 或 visual tokens，再通过 projector 对齐到语言模型空间，最后由 LLM 做统一推理和生成。

### 12.2 视觉编码器和语言模型如何衔接？

答：通常用线性层、MLP projector 或 cross-attention 把视觉特征映射到 LLM token embedding 空间。训练时让模型学习图文对齐。

### 12.3 图像 token 如何进入语言模型上下文？

答：图像经过视觉 encoder 得到一组 visual tokens，然后作为特殊 token 序列拼入文本上下文，LLM 对它们和文本一起 attention。

### 12.4 多模态 Agent 和纯文本 Agent 在工程上有什么区别？

答：多模态多了媒体上传、解析、存储、异步处理、审核、长耗时任务和更复杂的上下文压缩。控制面仍然需要 tool、memory、trace、安全。

### 12.5 多模态 Agent 如何处理图像、视频等长耗时任务？

答：使用异步 job、分阶段状态、进度通知、中间预览、失败重试和内容安全审核。视频任务尤其需要队列和资源调度。

## 13. Agent 训练与对齐

### 13.1 Agentic CPT、SFT、RL 三阶段训练分别是什么？

答：Agentic CPT 用大量工具/任务语料继续预训练通用能力；SFT 用人工或合成轨迹学习格式和行为；RL 用 reward 优化任务成功、安全和偏好。

### 13.2 为什么 SFT 时要 mask observation tokens？

答：observation 是环境返回，不是模型应该生成的目标。SFT 训练时应让模型学习 assistant 的 reasoning/action/final，而不是背 tool 返回内容。

### 13.3 tool call 轨迹数据如何构造训练样本？

答：样本包括 system、user、assistant tool_call、tool observation、assistant next action/final。训练目标通常只包含 assistant 部分，observation 作为条件输入。

### 13.4 Agent 训练和普通对话模型训练有什么区别？

答：Agent 训练强调工具选择、参数生成、多步状态、观察反馈和任务完成；普通对话更强调自然语言回复质量。Agent 数据必须包含环境交互轨迹。

### 13.5 为什么有些场景 SFT 效果不好？

答：可能因为数据质量差、任务分布变化、工具环境不稳定、只模仿表面格式、缺少负样本和验证信号。复杂决策只靠 SFT 很难学到长期收益。

### 13.6 reward model 适合解决什么问题？

答：reward model 适合评估偏好、任务完成质量、安全性和过程质量，用于 RL 或 reranking。它不适合替代确定性业务校验。
