# NanoHarness Component Index

This document is a cached structural index for frequently revisited NanoHarness architecture questions. It is meant to avoid repeated whole-repository scans when locating the runtime pieces behind tool execution, validation, permissions, sandboxing, and observability.

> Snapshot note: verify paths, signatures, and line numbers before making code changes if the repository has changed since this index was written.

## Tool Registry / router / wiring

- `agent_forge/tools/registry.py` ‚Äî `ToolRegistry`; main name-to-tool registry and execution boundary. Key methods: `register()`, `schemas()`, `execute()`, `_validate_arguments()`.
- `agent_forge/tools/tool_router.py` ‚Äî `ToolRouter`; filters available tool schemas per turn based on intent, risk/mode metadata, read-only wording, and active skill tool names.
- `agent_forge/runtime/wiring.py` ‚Äî `build_registry()`; constructs `WorkspaceSandbox` and registers built-in tools such as list/read/write/grep/patch/run/git/diagnostics/ask-human, plus MCP tools if configured.
- `agent_forge/tools/ask_human.py` ‚Äî declares a control signal only; direct execution fails closed and `AgentLoop` owns persistence/stop behavior.
- `agent_forge/tools/base.py` ‚Äî `Tool` base contract with `schema()` and `execute()`.
- `agent_forge/tools/mcp_config.py`, `agent_forge/tools/adapters/mcp_style_adapter.py`, `agent_forge/tools/mcp_stdio.py` ‚Äî MCP/external tool adapters into the local registry.
- `agent_forge/skills/registry.py` ‚Äî `SkillRegistry`; adjacent capability registry, not the core tool registry, but selected skills can widen router tool access through `tool_names`.

## Validator / schema validation / guardrails

- `agent_forge/tools/registry.py` ‚Äî `_validate_arguments()` and `_matches_type()` validate tool required args and primitive types before execution.
- `agent_forge/runtime/structured_output.py` ‚Äî `StructuredOutputParser`; extracts JSON from model text, validates a JSON-Schema-like subset, and builds repair prompts.
- `agent_forge/runtime/llm_client.py` ‚Äî `_parse_tool_calls()` validates provider tool-call arguments as JSON objects and `_tool_to_openai_schema()` converts local tool schemas.
- `agent_forge/safety/guardrails.py` ‚Äî `input_guardrail()`, `output_guardrail()`, and `tool_guardrail()` validate risky user input, final-answer claims, and unknown/repeated/malformed tool calls.
- `agent_forge/safety/command_policy.py` ‚Äî `check_command()` validates shell-like commands against deny/allow rules.
- `agent_forge/skills/registry.py` ‚Äî `SkillSpec.from_mapping()`, `_dict_field()`, `_list_field()` validate skill manifest fields.

## Permission / approval policy

- `agent_forge/safety/permission.py` ‚Äî `PermissionPolicy` and `PermissionDecision`; central allow/ask/deny decision layer. Reads are generally allowed, writes ask, commands are checked, network/delete/external actions denied.
- `agent_forge/safety/command_policy.py` ‚Äî command permission defense for `run_command`, blocking dangerous commands and allowlisting safe validation/read-only git inspection.
- `agent_forge/runtime/hooks.py` ‚Äî `ApprovalMode`, `PermissionHook`, and `HookManager.default()` connect permission policy to runtime hooks; modes include `trusted`, `on-write`, `on-risk`, `locked`, and `dry-run`.
- `agent_forge/runtime/agent_loop.py` ‚Äî constructs `HookContext`, runs hooks before tool execution, records `permission_check`, handles approval, and maps permission actions.
- `agent_forge/runtime/config.py` and `agent_forge/forge_cli.py` ‚Äî runtime/CLI approval knobs such as `approval_mode` and `auto_approve_writes`.
- Tool-level defense in `agent_forge/tools/write_file.py`, `agent_forge/tools/apply_patch.py`, and `agent_forge/tools/run_command.py` checks permission before side effects.

## Human input / continuation

- `agent_forge/runtime/human_input.py` ‚Äî `HumanInputStore`; atomic pending/responded/cancelled informational requests, stable ids, choice validation, and terminal-state protection.
- `agent_forge/runtime/agent_loop.py` ‚Äî intercepts pre-loop clarification and `ask_human`, records `waiting_human`, loads existing responses, and blocks cancelled requests.
- `agent_forge/forge_cli.py` ‚Äî `forge respond` records an answer or cancellation; `forge resume` injects the answered question into continuation context.
- `agent_forge/runtime/approval.py` remains separate because authorization and clarification have different semantics.

## Sandbox / workspace boundary / execution environment

- `agent_forge/safety/sandbox.py` ‚Äî `WorkspaceSandbox`; resolves paths under workspace root and blocks traversal, external paths, and secret-like files. Key methods: `resolve_path()`, `is_sensitive_path()`, `ensure_safe_path()`.
- `agent_forge/runtime/wiring.py` ‚Äî creates the shared sandbox and injects it into built-in tools.
- Tool sandbox usage in `agent_forge/tools/read_file.py`, `list_files.py`, `grep.py`, `write_file.py`, `apply_patch.py`, and `run_command.py`.
- `agent_forge/runtime/execution_environment.py` ‚Äî `ExecutionEnvironmentConfig` and `ExecutionEnvironment`; local, isolated git worktree, and constrained OCI modes with snapshot mounts, resource/network policy, command delegation, manifest/replay evidence, path validation, and redaction.
- `agent_forge/runtime/git_workspace.py` ‚Äî shared HEAD-relative binary diff/status layer, including untracked source files and excluding only untracked runtime-owned `.agent_forge` artifacts.
- `agent_forge/runtime/hooks.py` ‚Äî `ExecutionEnvironmentHook` enforces path/command restrictions before tools run.

## Observer / hooks / events / trace

- `agent_forge/observability/trace.py` ‚Äî `TraceRecorder`; main append-only run event recorder, metrics collector, and final trace JSON writer.
- `agent_forge/observability/event.py` ‚Äî `TraceEvent` typed event shape.
- `agent_forge/runtime/hooks.py` ‚Äî `RuntimeHook` and `HookManager`; pre-tool/post-tool/on-stop observer and extension mechanism, including permission, environment, and secret-redaction hooks.
- `agent_forge/runtime/agent_loop.py` ‚Äî central event emission points for task checkpoint, guardrail, context/tool routing, hook check, permission check, tool call, observation, and stop hooks.
- `agent_forge/observability/metrics.py`, `agent_forge/observability/usage_report.py`, and `agent_forge/observability/evidence.py` consume trace events for counters, human-readable usage reports, and citeable evidence.

## Multi-agent orchestration

- `agent_forge/multi_agent/coordinator.py` ‚Äî sequential Implementer/Reviewer/Verifier workflow with explicit artifact handoff and bounded revisions.
- `agent_forge/multi_agent/fanout.py` ‚Äî pure dependency batching and declared/actual overlap algorithms.
- `agent_forge/multi_agent/live_fanout.py` ‚Äî validated JSON plan, disposable worktree workers, fresh AgentLoop/LLM/registry instances, scope gate, deterministic merge, incremental checkpoint, selective recovery, and isolated finalizer.
- `examples/fanout-plan.sample.json` ‚Äî safe read-only two-worker plan for first execution.
